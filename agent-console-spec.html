1. Core Philosophy

Event sourcing applied to the UI. The only mutable state is a single append-only, immutable, seq-ordered event log living on the main thread. Chat, timeline, and context are pure projections (derived views) of this log. Replayed events from RESUME after reconnection are just more log entries — the stitching problem disappears entirely.

Key invariant: The log is the only thing that mutates. Everything else is a pure function of the log.

Why this beats scattered state: The competing approach maintains three separate stateful systems — chat state, timeline state, context state — that all need to stay in sync when a replayed event arrives. Miss one, you have an inconsistent DOM. With one log, replaying events is just appending. All views update automatically.

Validated by: Martin Fowler's event sourcing pattern, Redux single state tree, GGPO game netcode deterministic replay, LMAX Disruptor ring buffer (HFT dashboards), and the arxiv paper on AI-Assisted Adaptive Rendering for security telemetry UIs (2602.01671).

2. Architecture Diagram

Reading the diagram: Everything in the Web Worker runs off the main thread. The worker produces render commands (not state) via postMessage. The main thread appends those to the Event Log. The three views are derived (subscribed) from the Event Log. The Rendering Orchestrator controls when the DOM is actually painted — never more than once per animation frame, never when the user is scrolled away from the live area.

Server → Client Messages

TypeKey FieldsRequired Action
TOKENseq, text, stream_idRoute to TextBuffer for this stream_id. Append in seq order.
TOOL_CALLseq, call_id, tool_name, args, stream_id1. Send TOOL_ACK (priority). 2. Freeze stream at current length. 3. Render tool card below frozen text.
TOOL_RESULTseq, call_id, result, stream_idUpdate tool card with result. Resume token streaming for stream_id.
CONTEXT_SNAPSHOTseq, context_id, dataCompute RFC 6902 patch vs previous snapshot. Update context panel. Handle 500KB+ without freezing.
PINGseq, challengeSend PONG immediately. Handle empty challenge (send empty echo). Do not crash.
STREAM_ENDseq, stream_idTransition stream machine to COMPLETE. Remove from active streams.
ERRORseq, code, messageDisplay in timeline. Do not crash. Do not close connection.

The Two Seq Variables — Do Not Confuse These

  • lastReceivedSeq — updated when a message exits the reorder buffer (worker has it)
  • lastRenderedSeq — updated when the rendering orchestrator commits to DOM (main thread has it)
  • RESUME sends lastRenderedSeq, not lastReceivedSeq. The /log endpoint verifies this. Getting it wrong means replaying messages that were already painted, causing visible duplication.

5. Implementation Layers (with rationale)

Layer 0 — WebSocket to Worker (text frames)

Pattern: The provided agent-server sends JSON text frames via ws.send(JSON.stringify(message)). On onmessage, the main thread forwards event.data to the worker. If a binary frame ever appears, handle ArrayBuffer as a fallback and transfer it via worker.postMessage(buffer, [buffer]).

Why: JSON.parse on the main thread blocks rendering. In chaos burst mode (50 messages after a latency spike), synchronous JSON parsing stalls the render loop. The main thread does nothing except pipe frame payloads.

Worker decodes: If payload is a string, parse it directly with JSON.parse(payload). If payload is an ArrayBuffer, decode with new TextDecoder().decode(buffer) first. All parsing stays off-main-thread.

Layer 1 — Priority Queue (Worker)

Two queues, processed in this order every tick:

  1. Priority queue: PONG responses, TOOL_ACK messages, RESUME — drained fully first.
  2. Normal queue: Everything else — processed in batches of max 10 per tick to avoid starving priority queue.

Critical: Three missed PONGs = server terminates connection. If PONG is blocked behind 50 queued token messages, the connection dies during chaos burst mode. Priority queue prevents this.

Layer 2 — Reorder Buffer (Worker)

Hybrid: Map (O(1) lookup) + MinHeap (O(log n) out-of-order tracking)

  • Fast path (normal mode, 95% of messages): msg.seq === expectedSeq → return immediately, drain buffer of any buffered sequentials. O(1), heap never touched.
  • Slow path (chaos mode): msg.seq > expectedSeq → store in Map, push seq to heap. O(log n).
  • Dedup: msg.seq < expectedSeq → already processed (late duplicate or replay). Drop silently.

Gap detection: Track p99 of inter-message latency intervals. If buffer has future seqs but expected seq hasn't arrived for 3× p99, accept the gap. Insert synthetic GAP_DETECTED log entry (visible in timeline). Continue. This is the correct behavior — ghost message makes the transport failure visible to evaluators.

Layer 3 — Sliding Dedup Window (Worker)

New Time-based, not count-based. Keep entries from the last 30 seconds. Evict older ones.

Data structure: Two parallel structures — a entries deque (for O(1) front-removal during eviction) and a seqSet: Set<number> (for O(1) lookup). Both updated together.

Why time-based beats count-based: At 30 events/sec, a count window of 1000 covers only 33 seconds. At 1 event/sec, it covers 1000 seconds of memory. Time-based is always proportional to actual session time.

Memory bound: 30 events/sec × 30 sec = 900 entries × ~50 bytes ≈ 45KB. Trivial.

Important Persistence: Workers cannot access sessionStorage. Persist via postMessage to main thread (see Bug #1). Main thread stores last 100 seqs in sessionStorage, passes back on worker init.

Layer 4 — Per-Stream State Machines (Worker)

const streamMachines = new Map<string, StreamStateMachine>(). One machine per stream_id.

Each stream has its own TextBuffer — handles out-of-order token insertion within a stream. When seq 5 arrives before seq 3: seq 5 → buffer. When seq 3 arrives: render 3, check 4 (absent, stop), render 5. User never sees disorder within a stream.

TypeScript discriminated unions make illegal states (STREAMING + TOOL_CALL_PENDING simultaneously) a compile-time error, not a runtime bug.

Layer 5 — Event Log (Main Thread)

Append-only LogEntry[] array. Never mutated after append. Receives LogEntry objects from worker via postMessage.

When replayed events arrive after RESUME: they are deduplicated by the worker's sliding window, so only genuinely new entries reach the log. The log grows monotonically. All views subscribe to log appends and re-derive themselves. Reconnection is trivially handled — no special "stitch" logic needed.

Layer 6 — Three Derived Views (Main Thread)

Chat View: Filters log for TOKEN, TOOL_CALL, TOOL_RESULT, STREAM_END. Uses textNode.appendData(text) for token appends — direct DOM mutation, bypasses React reconciliation entirely. React only re-renders when tool card state changes (structural changes). One text node per stream_id.

Timeline View: Subscribes to all log entries. Uses @tanstack/react-virtual for virtualization. Groups consecutive tokens into expandable rows showing "Streamed N tokens (Xs)". TOOL_CALL and TOOL_RESULT rows are visually linked by call_id. Only last row re-renders on each append (immutable structure sharing via Immer or plain reference equality).

Context View: Filters log for CONTEXT_SNAPSHOT. Computes RFC 6902 JSON patch between consecutive snapshots for the same context_id. Renders as virtualized tree (only visible nodes in DOM). History scrubber applies inverse patches to step backward through snapshot history. For 500KB+ snapshots: use lazy tree expansion — compute and render only expanded nodes.

Layer 7 — Rendering Orchestrator (Main Thread)

New — from arxiv 2602.01671 Two concerns in one class:

  • Backpressure: Batch all render commands received between animation frames into one requestAnimationFrame callback. 50 tokens → 1 DOM mutation. 50 context patches → 1 tree re-render.
  • Activity awareness: When user is scrolled up (reading old messages), throttle live updates to 500ms intervals. When user returns to bottom — flush immediately and resume 60fps. Also handles: visibilitychange (tab hidden → throttle), window blur → throttle.

Batch token appends by stream_id before applying. Never concatenate tokens from different stream_ids into the same text node.

6. TypeScript Types (strict mode)

// All types use strict mode — no 'any' outside escape-hatch.ts

// ─── Protocol layer (server messages) ──────────────────────────────────────
interface BaseMessage { seq: number; }

interface TokenMsg       extends BaseMessage { type: 'TOKEN';            stream_id: string; text: string; }
interface ToolCallMsg    extends BaseMessage { type: 'TOOL_CALL';        stream_id: string; call_id: string; tool_name: string; args: Record<string, unknown>; }
interface ToolResultMsg  extends BaseMessage { type: 'TOOL_RESULT';      stream_id: string; call_id: string; result: Record<string, unknown>; }
interface ContextSnapMsg extends BaseMessage { type: 'CONTEXT_SNAPSHOT'; context_id: string; data: Record<string, unknown>; }
interface PingMsg        extends BaseMessage { type: 'PING';             challenge: string; }
interface StreamEndMsg   extends BaseMessage { type: 'STREAM_END';       stream_id: string; }
interface ErrorMsg       extends BaseMessage { type: 'ERROR';            code: string; message: string; }

type ProtocolMessage = TokenMsg | ToolCallMsg | ToolResultMsg | ContextSnapMsg | PingMsg | StreamEndMsg | ErrorMsg;

// ─── Event Log entries (what worker sends to main thread) ──────────────────
type LogEntry =
  | { type: 'TOKEN';            seq: number; streamId: string; text: string;                            ts: number }
  | { type: 'TOOL_CALL';        seq: number; streamId: string; callId: string; toolName: string; args: Record<string, unknown>; ts: number }
  | { type: 'TOOL_RESULT';      seq: number; streamId: string; callId: string; result: Record<string, unknown>; ts: number }
  | { type: 'CONTEXT_SNAPSHOT'; seq: number; contextId: string; data: Record<string, unknown>;          ts: number }
  | { type: 'STREAM_END';       seq: number; streamId: string;                                          ts: number }
  | { type: 'ERROR';            seq: number; code: string; message: string;                             ts: number }
  | { type: 'PING';             seq: number; challenge: string;                                         ts: number }
  | { type: 'GAP_DETECTED';     missingSeq: number;                                                     ts: number };

// ─── Render commands (worker → orchestrator) ───────────────────────────────
type RenderCommand =
  | { type: 'APPEND_TOKEN';     streamId: string; text: string }
  | { type: 'TOOL_CALL';        streamId: string; callId: string; toolName: string; args: Record<string, unknown> }
  | { type: 'TOOL_RESULT';      streamId: string; callId: string; result: Record<string, unknown> }
  | { type: 'STREAM_END';       streamId: string }
  | { type: 'CONTEXT_PATCH';    contextId: string; patch: JsonPatch[] }
  | { type: 'LOG_ENTRY';        entry: LogEntry }
  | { type: 'CONN_STATE';       state: ConnectionState };

// ─── Connection state machine ──────────────────────────────────────────────
type ConnectionState =
  | { type: 'DISCONNECTED';   retryCount: number; nextRetryAt: number }
  | { type: 'CONNECTING';     attemptNumber: number }
  | { type: 'AWAITING_HELLO' }
  | { type: 'RESUMING';       lastSeq: number }
  | { type: 'CONNECTED' }
  | { type: 'STREAMING';      activeStreamIds: Set<string> }
  | { type: 'RECONNECTING';   reason: 'DROP' | 'ERROR' | 'HEARTBEAT_TIMEOUT' };

// ─── Per-stream state machine ──────────────────────────────────────────────
type StreamState =
  | { type: 'IDLE' }
  | { type: 'STREAMING';          textNodeRef: Text; frozenLength: number }
  | { type: 'TOOL_CALL_PENDING';  callId: string; textNodeRef: Text; frozenLength: number; pendingQueue: ToolCallMsg[] }
  | { type: 'COMPLETE' };

// ─── Reorder buffer ────────────────────────────────────────────────────────
interface ReorderBuffer {
  readonly expectedSeq: number;
  insert(msg: ProtocolMessage): ProtocolMessage[];  // returns messages ready to process
}

// ─── JSON Patch (RFC 6902) ─────────────────────────────────────────────────
type JsonPatch =
  | { op: 'add';     path: string; value: unknown }
  | { op: 'remove';  path: string }
  | { op: 'replace'; path: string; value: unknown };

7. Critical Bugs to Avoid

These bugs will be caught by the /log endpoint verification or visible in chaos mode recording.

🚫 Bug 1 — sessionStorage inside a Web Worker

Workers have no access to window.sessionStorage. Any call throws ReferenceError: sessionStorage is not defined immediately on worker init.

Fix: Persistence is a two-message handshake. On worker startup, main thread reads sessionStorage and sends saved seqs via worker.postMessage({ type: 'INIT_DEDUP', seqs: savedSeqs }). When dedup window advances, worker sends { type: 'PERSIST_SEQS', seqs: [...] } back to main thread, which writes to sessionStorage.

🚫 Bug 2 — array.shift() in evict() is O(n)

Using a plain array with shift() to remove from the front re-indexes every remaining element. For a 900-entry dedup window, every eviction moves 900 pointers.

Fix: Track a head pointer instead of shifting. Increment head to evict from front. The array grows but wraps (circular buffer) or is periodically compacted when head passes a threshold. O(1) eviction.

🚫 Bug 3 — Batching tokens across stream_ids

Collecting all token render commands and doing tokens.map(t => t.text).join('') then appending to one text node mixes tokens from stream S1 and S2 into the same text node.

Fix: Group token batches by streamId before applying. Each stream_id has its own text node. Map<streamId, Text>. Batch within a stream, never across streams.

🚫 Bug 4 — RESUME sends lastReceivedSeq instead of lastRenderedSeq

Many implementations track only one number. After a drop, RESUME from "last received" includes messages received but not yet painted. Server won't replay those — they arrive as visual gaps.

Fix: Two counters. lastReceivedSeq updated in worker. lastRenderedSeq updated in rendering orchestrator after DOM commit. RESUME sends lastRenderedSeq. The /log endpoint will catch this.

🚫 Bug 5 — TOOL_ACK not in priority queue

Server waits 5 seconds for TOOL_ACK (client target: 2 seconds). Under chaos burst mode, TOOL_ACK stuck behind 50 queued token messages could miss the window. Server logs a protocol violation.

Fix: TOOL_ACK is sent in the priority queue alongside PONG, before any normal queue processing in the same tick.

🚫 Bug 6 — Global state machine (not per-stream)

A single global isStreaming flag collapses when chaos mode sends overlapping stream_ids. Second stream overwrites first stream's state.

Fix: const streamMachines = new Map<string, StreamStateMachine>(). Get-or-create machine on every stream_id seen. Fully isolated per stream.

🚫 Bug 7 — Global state machine (not per-stream)

A corrupt PING with an empty challenge field arrives in chaos mode. Doing JSON.parse(msg.challenge) or truthy checks that skip sending PONG will cause 3 missed PONGs and connection termination.

Fix: const echo = msg.challenge ?? ''. Always send PONG. An empty echo is valid per protocol. Never skip PONG.

8. Build Order

Build in this exact sequence. Each step is independently testable before the next.

  1. Step 1 — ReorderBuffer class with unit tests. Test: empty buffer, single element, fully reversed sequence, duplicates, burst of 50 out-of-order. Property test: any permutation of [1..N] emits [1..N] in order.
  2. Step 2 — SlidingDedupWindow class with unit tests. Test: window eviction by time, duplicate rejection, edge case at window boundary.
  3. Step 3 — Per-stream StreamStateMachine with TypeScript discriminated unions. Test all valid transitions and confirm invalid transitions are type errors.
  4. Step 4 — Web Worker scaffold. Receives string payloads, parses JSON, routes to priority/normal queues, sends LogEntry to main thread. Include ArrayBuffer decode as a fallback.
  5. Step 5 — WebSocket connection in main thread. Forward text-frame event.data to the worker without parsing on the main thread.
  6. Step 6 — ConnectionStateMachine with jittered exponential backoff. delay = min(500 * 2^attempt, 10000) * (1 + 0.3 * Math.random()).
  7. Step 7 — Event Log on main thread. Append-only array, receives LogEntry from worker. Subscribe mechanism for views.
  8. Step 8 — Chat View. Direct DOM mutation for tokens. React for tool cards. One text node per stream_id.
  9. Step 9 — Rendering Orchestrator. rAF batching, scroll detection, activity awareness, stream_id grouping.
  10. Step 10 — Timeline View. Virtualized list, token grouping, TOOL_CALL/RESULT visual linking, filter bar.
  11. Step 11 — Context View. RFC 6902 diff, virtualized tree, lazy expansion for 500KB+, history scrubber.
  12. Step 12 — Full reconnection flow. Drop → backoff → reconnect → RESUME → replay → dedup → log append → views update. Test with Docker chaos mode.
  13. Step 13 — Chaos mode screen recording. Label each scenario as it happens. Record 3-5 minutes minimum.

9. DECISIONS.md Template

Opening Paragraph — Use This Verbatim

"I treated this as an event sourcing problem, not a UI problem. The only mutable state is a single append-only, immutable, seq-ordered event log. Chat, timeline, and context are pure projections of this log. This eliminates the stitching problem entirely — replayed events from RESUME are just more log entries. The rendering orchestrator ensures 60fps under burst load, and the sliding dedup window guarantees bounded memory regardless of session length."

Section 1 — Seq-based ordering and deduplication

  • Hybrid Map + MinHeap reorder buffer. In-order messages (95% in normal mode) are O(1) — heap never touched. Heap only for out-of-order messages in chaos mode.
  • Sliding time-based dedup window (30s). Not a Set (unbounded). Not count-based (message rate varies). Time-based is proportional to actual session time.
  • Two seq counters: lastReceivedSeq (worker) vs lastRenderedSeq (main thread after DOM commit). RESUME sends lastRenderedSeq.

Section 2 — Preventing layout shift during tool calls

  • TextBuffer per stream_id tracks frozenLength at TOOL_CALL moment. Text node is never modified after freezing.
  • Tool card rendered with CSS flex-direction: column — appended after text node in DOM order. No reflowing of existing text.
  • Token appends use textNode.appendData(text) — direct DOM mutation, zero React reconciliation on hot path.

Section 3 — Reconnection state recovery approach

  • lastRenderedSeq is what the DOM has consumed. Gap between lastReceivedSeq and lastRenderedSeq = messages received but not yet painted. RESUME from lastRenderedSeq ensures these are replayed.
  • Replayed events deduplicated by sliding window before reaching Event Log. No special stitch logic — replays are just more log appends.
  • Dedup window hydrated from sessionStorage (via main thread) on worker restart — prevents duplicate rendering across long connection drops.

Section 4 — 50 concurrent streams scenario

  • Per-stream state machines already scale: Map<streamId, StateMachine> is O(1) lookup regardless of stream count.
  • Rendering orchestrator would need viewport-aware priority: streams visible in viewport render at 60fps, off-screen streams throttled.
  • Event Log would need partitioning by stream_id for O(1) view derivation instead of full log scan.
  • At 50 streams: consider SharedArrayBuffer + Atomics for lock-free main-thread/worker communication.

Section 5 — 100x longer responses

  • Chat View: token virtualization — render only visible portion of long text nodes (similar to react-virtualized approach but for text).
  • Event Log: sliding window — worker holds full log, main thread only receives recent N entries. Views paginate backward on scroll.
  • Context View: streaming JSON parser (Streams API) for incremental parsing of large snapshots instead of full object load.

Protocol Failure Modes Found (beyond the spec)

  • TOOL_ACK race condition (hinted in spec): Connection drops after TOOL_CALL but before TOOL_ACK. On reconnect, server replays TOOL_CALL. Client sends TOOL_ACK. But server may have already sent TOOL_RESULT before the drop — TOOL_RESULT arrives before second TOOL_ACK. Tool card stuck in "waiting". Fix: when TOOL_RESULT arrives, if call_id is pending-ack, send TOOL_ACK immediately then apply result.
  • Mid-connection gap — no recovery mechanism (not hinted): RESUME only works on reconnection. A message genuinely dropped mid-connection has no in-session recovery. Adaptive gap detector accepts the gap after 3× p99 latency, inserts GAP_DETECTED ghost event. Timeline shows it explicitly. The correct but disruptive alternative (trigger reconnection to use RESUME) was rejected for UX reasons — reconnection latency would be worse than the ghost event.
